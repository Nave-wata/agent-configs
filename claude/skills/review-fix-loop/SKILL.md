# レビュー→修正ループ

レビューと修正を繰り返し、Critical/Important の指摘がなくなるまで自動でループする。

## コンテキスト

- 現在のブランチ: !`git branch --show-current`
- git status: !`git status --short`
- 変更ファイル: !`git diff --name-only HEAD~1 2>/dev/null || git diff --name-only`
- 最近のコミット: !`git log --oneline -5`

## パラメータ解析

$ARGUMENTS から以下を抽出する:

1. **Issue番号**（必須）: 数字または GitHub Issue URL から抽出。取得できない場合はユーザーに確認する
2. **レビュー対象**（任意）: Issue番号の後の残りの引数。指定がなければデフォルト値を使用（後述）

## 設定

| 項目 | 値 |
|------|---|
| 最大ループ回数 | 5 |
| 自動コミット | 有効（`/commit` スキルに委譲） |
| 終了条件 | Fix 分類の Critical/Important が両方 0 件 |

## 実行手順

### 0. 事前準備

#### ワークツリーのクリーン確認

`git status` で未コミットの変更を確認する。

- **変更なし**: 続行
- **変更あり**: ユーザーに確認する
  - 「先にコミットする」→ `/commit` を実行してから続行
  - 「stash する」→ `git stash` してから続行
  - 「中断する」→ 処理を停止

#### レビュー対象の基準を固定

ループ全体で一貫したレビュー対象を使用するため、開始時に基準を確定する:

- **引数でレビュー対象が指定された場合**: そのまま使用（例: `main...HEAD`）
- **引数で指定がない場合**: `BASE_SHA` を確定する
  - 作業ブランチの場合: `git merge-base main HEAD` で分岐点を取得し、`{BASE_SHA}...HEAD` を使用
  - main ブランチの場合: `HEAD~1` を使用

この基準は全イテレーションで共通して使用する。修正コミット後も常にこの基準で差分を取るため、未解決の指摘がレビューから外れることはない。

### ループ開始（最大5回）

各イテレーションで以下を実行する。ループカウンタ `N` を 1 から開始。

**重要**: レビュー・修正はサブエージェント（Agent ツール）で実行し、メインエージェントのコンテキストを節約すること。メインエージェントの役割はオーケストレーション（結果評価、ループ制御）に限定する。

---

### N-1. レビュー実行（サブエージェント）

Agent ツールでレビュー用サブエージェントを起動する。

サブエージェントへの指示:
- Skill ツールで `/review` スキルを呼び出してレビューを実行すること
- レビュー対象の差分基準: `{ステップ0で確定した基準}`
- 2回目以降: Won't Fix リストの内容を渡し、該当する指摘は除外するよう指示
- `/review` の結果を構造化して返すこと（Critical / Important / Suggestions に分類）

サブエージェント内で `/review` スキルが pr-review-toolkit エージェント群と Codex MCP を並列起動し、結果を統合して返す。

#### レビュー結果の後処理（メインエージェント）

サブエージェントから返された結果に、一意のIDを振る:
- Critical Issues → [C1], [C2], ...
- Important Issues → [I1], [I2], ...
- Suggestions → [S1], [S2], ...

2回目以降は、Won't Fix リストに該当する指摘を結果から除外する（同一箇所・同一内容の指摘）。
ただし、修正によって状況が変わって再指摘されたものは新規指摘として扱う。

### N-2. 結果評価（メインエージェントが実施）

**全レベルの指摘**（Critical / Important / Suggestions）を評価対象とし、各指摘を以下のいずれかに分類する:

| 分類 | 説明 | アクション |
|------|------|-----------|
| **Fix** | 修正が必要 | 修正エージェントに渡す |
| **Won't Fix** | 仕様として許容 / 誤検知 | Won't Fix リストに追加し、理由を記録 |
| **Defer** | 今回のスコープ外 | 記録のみ |

判断基準:
- コードの意図・仕様として妥当な指摘は Won't Fix
- 既存コードのスタイルや慣習に沿っているものは Won't Fix
- 明らかなバグ、セキュリティリスク、パフォーマンス問題は Fix
- Suggestions でも改善効果が明確なものは Fix に分類する
- **判断に迷う場合はユーザーに確認する**

#### 終了判定

Fix に分類された **Critical / Important** が 0 件 → **ループ終了**(ステップ F へ)

※ Suggestions のみ Fix が残っている場合もループは終了する。ただし最終イテレーションまでに Fix 分類された Suggestions は修正対象に含める。

### N-3. 修正実行（サブエージェント）

Agent ツールで修正用サブエージェントを起動する。`mode: "acceptEdits"` を指定。

サブエージェントへの指示:
- Fix に分類された指摘のリスト（ID、説明、対象ファイル:行番号）
- 各指摘に対するメインエージェントからの補足・方針（あれば）
- Won't Fix の理由（修正エージェントが同じ箇所を変更しないよう）
- 修正のスコープを厳守すること（指摘された問題のみ修正、周辺のリファクタリングは禁止）

### N-4. 修正後のコミット

修正エージェントの完了後:

1. `git diff` で変更内容を確認し、意図しない変更がないか検証する
2. Skill ツールで `/commit` スキルを呼び出す（Issue番号を引数に渡す）

### N-5. ループ継続判定

- ループカウンタ N < 最大ループ回数 → N を +1 して **N-1** に戻る
- ループカウンタ N >= 最大ループ回数 → **ループ上限到達**（ステップ F へ）

---

## 異常系ハンドリング

各ステップで以下の異常が発生した場合の対応:

| 状態 | 対応 |
|------|------|
| **review_failed**: レビューサブエージェントが想定フォーマットで結果を返さない | メインエージェントが結果を手動で解析・正規化して続行。解析不能な場合はユーザーに報告して中断 |
| **fix_no_diff**: 修正サブエージェントが差分を出さない（修正できなかった） | 該当指摘をユーザーに報告し、手動対応を依頼。残りの指摘があればループ続行 |
| **commit_failed**: `/commit` が失敗する | エラー内容をユーザーに報告して中断。変更はワークツリーに残る |
| **max_loop_reached**: 最大ループ回数に到達 | 残っている Critical/Important をユーザーに報告し、対応方針を確認 |

---

### F. 最終報告

ループ完了後、以下を報告する:

```markdown
# レビュー→修正ループ 完了報告

## 結果サマリー
- 実行回数: N / 最大5
- 終了理由: （指摘解消 / ループ上限到達）

## 修正内容（イテレーションごと）
### イテレーション 1
- [C1] xxx → 修正済み（コミット: abc1234）
- [I1] yyy → 修正済み（コミット: abc1234）

### イテレーション 2
- ...

## Won't Fix（仕様として許容）
- [I3] zzz — 理由: ...

## Deferred（スコープ外）
- [S2] www — 理由: ...

## 未解決の指摘（ループ上限到達時のみ）
- ...

## コミット履歴
- abc1234: #issue [fix]: ...
- def5678: #issue [fix]: ...
```

ループ上限に到達して未解決の Critical/Important が残っている場合は、ユーザーに残りの対応方針を確認する。

## Won't Fix リストの管理

Won't Fix リストはループ全体で蓄積する。各エントリは以下の情報を持つ:

```
- ID: [C1], [I2] など
- 指摘内容: レビューエージェントの指摘文
- 理由: Won't Fix と判断した理由
- 対象: file:line
```

2回目以降のレビュー結果評価時に、Won't Fix リストと照合して同じ指摘を除外する。
ただし、修正によって状況が変わった場合は再度指摘される可能性がある（これは正常な動作）。
