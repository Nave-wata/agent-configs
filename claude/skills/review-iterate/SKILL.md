# 対話型レビュー反復（review-iterate）

レビューと修正を繰り返し、Critical/Important の指摘がなくなるまでループする。
ただし **各イテレーションの修正はユーザーと相談しながら対話的に行う**。`review-fix-loop` との違いは「自動修正・自動コミットではなく、ユーザー承認ベースで進める」点。

## 🚫 厳守事項（リモート影響の禁止）

このスキルは **ローカルの作業ツリーにのみ影響を与える**。以下を**絶対に**実行しないこと:

- `git commit` / `git push` / `git tag` / `git stash`（破壊的操作）
- `gh pr create` / `gh pr merge` / `gh pr review` / `gh issue` 系の **書き込み**操作
- `gh` / `curl` でのラベル更新・コメント投稿・ステータス変更など**リモートを変更する一切の操作**
- `/commit`・`/create-pr`・`/release` 等のコミット/リリース系スキルの呼び出し

許可されるのは、レビューのための**読み取り**（`git diff`/`gh pr view` 等）と、**作業ツリー上のファイル編集**（Edit/Write）のみ。
修正結果は未コミットの作業ツリー変更として残す。コミットが必要かどうかは、ループ完了後にユーザーが別途判断する（このスキルでは行わない）。

## コンテキスト

- 現在のブランチ: !`git branch --show-current`
- git status: !`git status --short`
- 変更ファイル: !`git diff --name-only HEAD~1 2>/dev/null || git diff --name-only`
- 最近のコミット: !`git log --oneline -5`

## パラメータ解析

$ARGUMENTS から **レビュー対象**（任意）を抽出する。指定がなければデフォルト基準を使用（後述）。
Issue 番号は不要（コミットしないため）。

## 設定

| 項目 | 値 |
|------|---|
| 最大ループ回数 | 5 |
| コミット | **行わない**（作業ツリーのみ） |
| 修正の進め方 | ユーザーと相談・承認ベース |
| 終了条件 | Critical/Important の未解決指摘が 0 件 |

## 実行手順

### 0. 事前準備

#### レビュー対象の基準を固定

ループ全体で一貫したレビュー対象を使うため、開始時に基準を確定する。修正後もこの基準で差分を取るため、未解決の指摘がレビューから外れない。

- **引数で指定された場合**: そのまま使用（例: `main...HEAD`）
- **指定がない場合**: `BASE_SHA` を確定する
  - 作業ブランチの場合: `git merge-base main HEAD` で分岐点を取得し `{BASE_SHA}...HEAD` を使用
  - main ブランチの場合: `HEAD~1` を使用

未コミットの作業ツリー変更も含めてレビューしたい場合があるため、各イテレーションのレビューでは「確定した基準の差分」＋「未コミットの作業ツリー差分」の両方を対象とする。

---

### ループ開始（最大5回）

各イテレーションで以下を実行する。ループカウンタ `N` を 1 から開始。

**重要**: レビューはサブエージェント（Agent ツール）で実行し、メインエージェントのコンテキストを節約する。メインエージェントの役割はオーケストレーション（結果提示・ユーザーとの相談・ループ制御）と、ユーザー承認後の修正適用に限定する。

---

### N-1. レビュー実行（サブエージェント）

Agent ツールでレビュー用サブエージェントを起動する。

サブエージェントへの指示:
- Skill ツールで `/review` スキルを呼び出してレビューを実行すること
- レビュー対象の差分基準: `{ステップ0で確定した基準}`（未コミットの作業ツリー変更も含める）
- 2回目以降: これまでに確定した **Won't Fix / Defer リスト**を渡し、該当する指摘は除外するよう指示
- `/review` の結果を構造化して返すこと（Critical / Important / Suggestions に分類、各指摘に file:line を付与）

サブエージェント内で `/review` が pr-review-toolkit エージェント群と Codex MCP を並列起動し、結果を統合して返す。

#### レビュー結果の後処理（メインエージェント）

返された結果の各指摘に、イテレーションをまたいで一意のIDを振る:
- Critical → [C1], [C2], ...
- Important → [I1], [I2], ...
- Suggestions → [S1], [S2], ...

2回目以降は Won't Fix / Defer リストに該当する指摘を除外する。ただし修正により状況が変わって再指摘されたものは新規指摘として扱う。

### N-2. ユーザーへの結果提示と相談（メインエージェント）

レビュー結果をユーザーに提示する。以下のフォーマットで簡潔にまとめる:

```markdown
# イテレーション N レビュー結果

## Critical Issues（修正必須）
- [C1] 問題の説明 [file:line]（検出元: agent名/Codex）

## Important Issues（修正推奨）
- [I1] 問題の説明 [file:line]（検出元: agent名/Codex）

## Suggestions（検討事項）
- [S1] 提案内容 [file:line]（検出元: agent名/Codex）
```

提示後、各指摘の対応方針を **AskUserQuestion でユーザーに相談する**。判断材料（影響範囲・推奨対応）を添える。各指摘を以下に分類:

| 分類 | 説明 | アクション |
|------|------|-----------|
| **Fix** | 今回修正する | ユーザー承認後に修正適用 |
| **Won't Fix** | 仕様として許容 / 誤検知 | リストに追加し理由を記録 |
| **Defer** | 今回のスコープ外（後日対応） | リストに記録のみ |

- まとめて相談してよいが、Critical は必ず個別に対応可否を確認する
- ユーザーの指示・追加方針（修正の具体的なやり方など）はそのまま修正に反映する
- 判断に迷う指摘は必ずユーザーに確認する

### N-3. 修正適用（メインエージェント、ユーザー承認済みのみ）

Fix に分類された指摘のみを、ユーザーと合意した方針に従って **作業ツリー上で** 修正する（Edit/Write）。

- 修正のスコープを厳守する（指摘された問題のみ。周辺の無関係なリファクタリングは禁止）
- Won't Fix とした箇所は変更しない
- **コミットはしない**。変更は未コミットのまま残す
- 修正規模が大きい場合や独立性が高い場合は、Agent ツールの修正サブエージェント（`mode: "acceptEdits"`）に委譲してよい。その際もコミット禁止・スコープ厳守を明示する
- 修正後、`git diff` で意図しない変更が混入していないか確認する

### N-4. 終了判定とループ継続

修正適用後、終了条件を評価する:

- **Fix 対象だった Critical/Important がすべて解消**（＝未解決の Critical/Important が 0 件）→ **ループ終了**（ステップ F へ）
  - ※ Suggestions のみが残っている場合もループは終了してよい。残った Suggestions は Defer として記録する
- 未解決の Critical/Important が残る、かつ N < 5 → N を +1 して **N-1**（再レビュー）に戻る
  - 再レビューで修正が正しく反映されたか、新たな問題が生じていないかを検証する
- N >= 5（最大ループ到達）→ **ループ上限到達**（ステップ F へ）

ユーザーが途中で「ここで止める」と判断した場合は、その時点でステップ F へ移る。

---

## 異常系ハンドリング

| 状態 | 対応 |
|------|------|
| **review_failed**: レビューサブエージェントが想定フォーマットで返さない | メインエージェントが結果を手動で正規化して続行。解析不能ならユーザーに報告して中断 |
| **fix_blocked**: 修正方針が定まらない / 影響が大きい | ユーザーに判断を仰ぐ。合意できなければ Defer として記録し次へ |
| **no_progress**: 同じ指摘が修正後も解消しない | 修正が不十分な可能性をユーザーに報告し、方針を再相談 |
| **max_loop_reached**: 最大ループ到達 | 残る Critical/Important をユーザーに報告し、対応方針を確認 |

---

### F. 最終報告（複数イテレーションの総合結果）

ループ完了後、全イテレーションを横断した総合結果を報告する:

```markdown
# 対話型レビュー反復 完了報告

## 結果サマリー
- 実行回数: N / 最大5
- 終了理由: （Critical/Important 解消 / ユーザー判断で終了 / ループ上限到達）
- 変更状態: 作業ツリーに未コミットの変更あり（コミット/push は未実施）

## イテレーションごとの経緯
### イテレーション 1
- [C1] xxx [file:line] → 修正済み（方針: ...）
- [I1] yyy [file:line] → 修正済み
### イテレーション 2
- ...

## Won't Fix（仕様として許容 / 誤検知）
- [I3] zzz [file:line] — 理由: ...

## Deferred（今回スコープ外）
- [S2] www [file:line] — 理由: ...

## 未解決の指摘（ループ上限到達時のみ）
- [C2] ... — 残存理由 / 推奨対応

## 変更ファイル一覧（作業ツリー）
- git diff --name-only の結果
```

報告後、変更は**未コミットのまま**であることを明示し、コミットするかどうかはユーザーに委ねる（このスキルではコミットしない）。
ループ上限到達で未解決の Critical/Important が残る場合は、残りの対応方針をユーザーに確認する。

## Won't Fix / Defer リストの管理

両リストはループ全体で蓄積する。各エントリは以下を持つ:

```
- ID: [C1], [I2] など
- 指摘内容: レビューエージェントの指摘文
- 分類: Won't Fix / Defer
- 理由: そう判断した理由（ユーザーとの合意内容）
- 対象: file:line
```

2回目以降のレビュー結果評価時にこのリストと照合し、同じ指摘を除外する。
ただし修正により状況が変わった場合は再指摘されることがある（これは正常な動作）。
